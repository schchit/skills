# Learning Rules

## What to Learn

### High-value signals (always learn)
- User explicitly corrects your behavior → immediately distill into a rule
- User confirms a non-obvious approach → stable preference
- The same pattern repeats across 3+ interactions → stable convention
- User explicitly states a project rule ("we always use X for Y") → project convention

### Medium-value signals (learn after repetition)
- `self-improving-prompt` version choice preference (refined vs original) — only record after the pattern becomes stable
- Output-format preferences (list vs paragraph, concise vs detailed)
- Tool-usage preferences (prefer / avoid certain tools)

### Low-value signals (usually skip)
- One-off debugging choices
- Temporary workarounds
- Decisions tightly bound to a single scenario and hard to reuse

## What NOT to Learn

### Never store
- Full prompt text generated by `self-improving-prompt`
- API keys, passwords, tokens, or any secret
- Personal sensitive information
- Highly changeable concrete file paths (prefer patterns over paths)
- Function signatures (derivable from code)

### Skip unless explicitly required
- Obvious programming common sense
- Framework defaults (for example, "use async/await in Node")
- Anything already clearly documented in README or config

## How to Judge self-improving-prompt Preference Signals

| Pattern | Action |
|--------|--------|
| User chooses refined version 5+ times in a row | Record: "user prefers refined prompts" |
| User usually chooses refined version but occasionally picks original | Don't overfit; likely scenario-specific |
| User says "stop showing me the original" | Record immediately as a rule |
| User says "I always want to see both options" | Record immediately as a rule |
| User switches between refined and original fairly evenly | Don't record (unstable preference) |

## Merge Strategy Examples

### Add a new rule
```
Before: (No testing strategy in CLAUDE.md)
Signal: User says "Always write tests before implementation"
After:  - Test first: write tests before implementation [tentative]
```

### Update a conflicting rule (correction marker)
```
Before: - Use AskUserQuestion popup confirmation
Signal: User says "Don't use popups, just show it in chat"
After:
  - ~~Use AskUserQuestion popup confirmation~~ [corrected: 2026-04-10]
  - Use plain-text confirmation, not AskUserQuestion popup [stable]
```

### Rule upgrade (tentative → stable)
```
Before: - Test first: write tests before implementation [tentative]
Signal: User confirms this approach again in a later session
After:  - Test first: write tests before implementation [stable]
```

### Merge redundant rules
```
Before:
- Don't record full prompt text
- Don't record the exact full prompt text produced by self-improving-prompt
- Only learn user preference rules, not full prompt content

After:
- Only learn preference rules, never record full prompt text
```
