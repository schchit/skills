## 核心设计原则

### 1. 用户驱动（不是自动化驱动）
- ❌ **之前**：直接打开浏览器（生硬）
- ✅ **现在**：提示用户选择，用户明确指令后再执行

### 2. 多轮交互（不是一次性返回）
- ❌ **之前**：单轮返回预约流程 + 自动打开
- ✅ **现在**：多轮对话，逐步引导完成预约

### 3. 意图识别（而不是硬编码）
```javascript
detectIntent("打开链接") → "open"
detectIntent("帮我预约") → "book"
detectIntent("咨询客服") → "consult"
```

### 4. 平台无关（打开浏览器跨平台）
```javascript
macOS:  open "url"
Windows: start "url"
Linux:  xdg-open "url"
```

### 5. 错误优雅降级
- 自动打开浏览器失败 → 返回 URL 让用户手动打开
- 自动点击按钮失败 → 提示用户手动点击位置

## 四轮完整对话流程

```
第1轮 (VIEW)
用户: "怎么预约 CNP 皮肤科？"
Skill: 
  [预约流程说明...]
  💡 接下来你可以：
  • "打开链接"
  • "帮我预约"
  • "咨询客服"

第2轮 (OPEN)
用户: "打开链接"
Skill:
  ✅ 链接已打开！
  [页面可见内容...]
  💡 接下来你可以：
  • "帮我预约"
  • "需要截图"

第3轮 (BOOK)
用户: "帮我预约"
Skill:
  ✅ 已点击预约按钮！
  📝 请填写以下信息：
  [表单字段指南...]

第4轮 (可选)
用户: "咨询客服"
Skill:
  📞 客服联系方式...
```

## 技术栈

| 组件 | 技术 | 作用 |
|------|------|------|
| 意图识别 | 关键词匹配 | 解析用户指令 |
| 浏览器操作 | `child_process.exec()` | 打开系统浏览器 |
| 自动化 | Playwright | 点击元素、导航 |
| 医院匹配 | 四级模糊匹配 | 从查询词找医院 |
| 模板渲染 | 自定义渲染器 | 生成预约流程文案 |
| 多语言 | i18n JSON | 支持 zh/en/ja/th |

## 常见问题

**Q: 为什么不直接自动化所有步骤？**
A: 因为用户需要控制权。直接全自动可能太激进，而多轮对话让用户在每一步都可以选择、暂停或改变主意。

**Q: 如果用户多次调用"帮我预约"会发生什么？**
A: 每次都会尝试打开浏览器和点击按钮。现在没有状态保存，所以不知道之前已经点过。这可以通过 context 改进。

**Q: Playwright 会留下浏览器窗口吗？**
A: 是的，`headless: false` 表示显示浏览器窗口，方便用户看到发生了什么。关闭 Playwright 进程后浏览器窗口保留。

**Q: 支持其他语言吗？**
A: 现在的对话文本是硬编码的中文。需要通过 i18n 系统翻译所有提示文本。
