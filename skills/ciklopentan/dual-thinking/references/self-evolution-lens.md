# Self-Evolution Lens
#tags: skills review

Version note: aligned with Dual Thinking v8.5.8 release line.

## Purpose
Use this lens when `dual-thinking` reviews itself or a skill in its native review/orchestrator/problem-framing domain.
The goal is not acceptability.
The goal is strength-for-purpose.
This reference follows the binding `Self-evolution lock` and `Current-date native-purpose maximization lock` in `SKILL.md`; it does not create parallel runtime law.

## Trigger conditions
Activate the self-evolution lens when ANY of these are true:
- the artifact under review is `dual-thinking`
- the user asks `dual-thinking` to improve itself
- the artifact is a skill in the same native domain as `dual-thinking`
- the user asks for stronger self-review, meta-review, self-hardening, or self-rewrite
- the user asks how to make the skill more powerful for the purpose it was created for

Precedence: early-exit conditions in `When not to use dual-thinking` suppress the lens for trivial work.

## Outside-self stance
When triggered:
- reason as if the current artifact were written by someone else
- act like a replacement architect, hostile maintainer, stronger competing skill designer, or future successor trying to obsolete the current artifact
- do not protect the current design merely because it is current
- do not treat prior publication, validation, or freeze status as proof that current wording is still optimal

## Strength-for-purpose questions
Ask in substance:
- where would a stronger current-date successor beat this version for its native mission?
- what should be removed, tightened, compressed, re-ordered, or structurally replaced?
- what part still exists to preserve history rather than increase capability?
- what domain-specific current patterns would materially strengthen this artifact?
- what is the smallest stronger patch, or if that is unsafe in the current line, what explicit structural deferral is required?

## Patch discipline
- prefer the smallest patch that materially increases native strength
- do not add ornamental philosophy
- if the current line is frozen and the change is structural, defer honestly to the next candidate line

## Weak-model honesty
Compact form is allowed, but it must still preserve:
- one real outside-self weakness or bottleneck
- one real smallest stronger patch or explicit justified structural deferral
- one real synthesis decision grounded in strength-for-purpose rather than loyalty-to-current-form
