# Dating CLI Detailed Operations

This document is for AI Agents using the `dating` skill.  
It explains `dating-cli` command usage, input fields, and output fields.

## 1. Basic Conventions

- CLI name: `dating-cli`
- Local config file: `~/.dating-cli/config.json`
- Output format: JSON by default, suitable for machine parsing

## 2. Common Rules

### 2.1 Parameter Input Rules

- Use `--help` on each command to see supported options.
- Most commands return a standard JSON envelope.

### 2.2 Error Output Structure

```json
{
  "ok": false,
  "error": {
    "message": "...",
    "status": 500,
    "payload": {}
  }
}
```

## 3. Success Output Structure

Most successful commands return:

```json
{
  "ok": true,
  "response": {
    "code": 200,
    "message": "Success",
    "data": {}
  }
}
```

Common extra top-level fields:

| Field | Type | Description |
|---|---|---|
| ok | Boolean | Whether the command succeeded |
| savedTo | String | Config file path where updates were saved |
| configPath | String | Local config path (`config` commands) |
| config | Object | Local config snapshot (`config` commands) |
| taskState | Object | Latest task snapshot (`task` commands) |

## 4. Config Commands

### 4.1 `dating-cli config show`

Purpose: Show current local config without calling backend APIs.

Input fields:

| Field | Type | Required | Description |
|---|---|---|---|
| (none) | - | - | No arguments |

Output fields:

| Field | Type | Description |
|---|---|---|
| ok | Boolean | Always `true` |
| configPath | String | Absolute local config file path |
| config.token | String | JWT token (may be absent) |
| config.tokenHead | String | Token prefix (may be absent) |
| config.memberId | Long | Current logged-in memberId (may be absent) |
| config.username | String | Current username (may be absent) |
| config.updatedAt | String | Last update time (ISO datetime) |

## 5. Account and Auth Commands

### 5.1 `dating-cli register [--username <username>]`

Mapped API: `POST /register`

Input fields:

| Field | Type | Required | Description |
|---|---|---|---|
| --username | String | No | Preferred username; server auto-generates if omitted |
| --password | String | No | Login password |

Output fields:

| Field | Type | Description |
|---|---|---|
| ok | Boolean | Command success |
| savedTo | String | Local config save path |
| response.code | Integer | Business status code |
| response.message | String | Business message |
| response.data.memberId | Long | New account memberId |
| response.data.username | String | Username |
| response.data.password | String | Initial password generated by server |
| response.data.token | String | Login token |
| response.data.tokenHead | String | Token prefix |

### 5.2 `dating-cli login --username <username> --password <password>`

Mapped API: `POST /login`

Input fields:

| Field | Type | Required | Description |
|---|---|---|---|
| --username | String | Yes | Login username |
| --password | String | Yes | Login password |

Output fields:

| Field | Type | Description |
|---|---|---|
| ok | Boolean | Command success |
| savedTo | String | Local config save path |
| response.code | Integer | Business status code |
| response.message | String | Business message |
| response.data.memberId | Long | Account memberId |
| response.data.username | String | Username |
| response.data.token | String | Login token |
| response.data.tokenHead | String | Token prefix |

### 5.3 `dating-cli logout [--token <token>] [--token-head <head>]`

Mapped API: `POST /logout`

Output fields:

| Field | Type | Description |
|---|---|---|
| ok | Boolean | Command success |
| savedTo | String | Local config save path after clearing auth |
| config | Object | Cleared local config |
| response.code | Integer | Business status code |
| response.message | String | Business message |
| response.data | Object | API payload (may be empty) |

## 6. Profile Command

### 6.1 `dating-cli upload <filePaths...> [--field-name <name>]`


Input fields:

| Field | Type | Required | Description |
|---|---|---|---|
| filePaths | String[] | Yes | One or more local file paths to upload |
| --field-name | String | No | Multipart field name, default `file` |

Output fields:

| Field | Type | Description |
|---|---|---|
| ok | Boolean | Command success |
| photoUrls | Array<String> | Uploaded image URLs (deduplicated) |

| profileUpdateResponse.message | String | Business message for profile update |
| profileUpdateResponse.data | Object | Updated profile payload |

| --ability-text | String | No | Ability text |
| --major | String | No | Major |
| --nationality | String | No | Nationality |
| --country | String | No | Country |
| --province | String | No | Province/State |
| --city | String | No | City |
| --address-detail | String | No | Address detail |
| --current-latitude | Number | No | Current latitude |
| --current-longitude | Number | No | Current longitude |
| --current-location-text | String | No | Current location text |
| --photo-url | String | No | Profile image URL, repeatable; maps to `photoUrls` |
| --email | String | No | Email used for match reminder notifications |
| --phone | String | No | Phone number |
| --telegram | String | No | Telegram |
| --wechat | String | No | WeChat |

## 7. Task Commands

### 7.1 `dating-cli task create --task-name <name> [criteria options]`

Mapped API: `POST /match-tasks`

Controller behavior: returns `CommonResult<MatchTaskResponse>`.

Input fields (base):

| Field | Type | Required | Description |
|---|---|---|---|
| --task-name | String | Yes | Task name |

Input fields (criteria):

| Field | Type | Required | Description |
|---|---|---|---|
| --preferred-gender-filter | String(JSON object) | No | Gender filter, e.g. `{"eq":"female"}` |
| --preferred-height-filter | String(JSON object) | No | Height filter, e.g. `{"gte":165,"lte":180}` |
| --preferred-income-filter | String(JSON object) | No | Income filter |
| --preferred-city-filter | String(JSON object) | No | City filter |
| --preferred-nationality-filter | String(JSON object) | No | Nationality filter |
| --preferred-education-filter | String(JSON object) | No | Education filter |
| --preferred-occupation-filter | String(JSON object) | No | Occupation filter |
| --preferred-education-stage | String | No | Education keyword |
| --preferred-occupation-keyword | String | No | Occupation keyword |
| --preferred-hobby-text | String | No | Preferred hobby text |
| --preferred-character-text | String | No | Preferred personality text |
| --preferred-ability-text | String | No | Preferred ability text |
| --intention | String | No | Match intention text, e.g. long-term relationship |
| --hobby-embedding-min-score | Number | No | Min semantic similarity score for hobby matching (usually leave unset; task create default `0.0` when omitted, recommended to use `0.0`) |
| --character-embedding-min-score | Number | No | Min semantic similarity score for personality matching (usually leave unset; task create default `0.0` when omitted, recommended to use `0.0`) |
| --ability-embedding-min-score | Number | No | Min semantic similarity score for ability matching (usually leave unset; task create default `0.0` when omitted, recommended to use `0.0`) |
| --intention-embedding-min-score | Number | No | Min semantic similarity score for intention matching (usually leave unset; task create default `0.0` when omitted, recommended to use `0.0`) |
| --preferred-contact-channel | String | No | Preferred channel: `phone/telegram/wechat/signal_chat/line/snapchat/instagram/facebook` |

Output fields (top-level):

| Field | Type | Description |
|---|---|---|
| ok | Boolean | Command success |
| savedTo | String | Local config save path |
| taskState.taskId | Long | Latest task ID |
| taskState.taskName | String | Latest task name |
| taskState.status | String | Latest task status (`active/stopped`) |
| taskState.matchStatus | String | Latest match status |

## 8. Match Check Command

### 8.1 `dating-cli check <taskId> [--page <value>]`

Input fields:

| Field | Type | Required | Description |
|---|---|---|---|
| taskId | Integer | Yes | Task ID |
| --page | Integer | No | Page number, default `1`; each page returns `10` candidates |

Output fields (top-level):

| Field | Type | Description |
|---|---|---|
| ok | Boolean | Command success |
| response.code | Integer | Business status code |
| response.message | String | Business message |
| response.watchStatus | String | Top-level `CommonResult.watchStatus` from controller |
| response.serverTime | Long | Top-level `CommonResult.serverTime`; currently may be `null` |
| response.data | Object | Check result object (see below) |

Output fields (`response.data`):

| Field | Type | Description |
|---|---|---|
| watchStatus | String | `MATCH_FOUND` or `NO_RESULT_RETRY_NOW`; CLI normalizes it to match top-level `response.watchStatus` |
| serverTime | Long | Server timestamp in milliseconds; currently may be `null`, and CLI normalizes it to match top-level `response.serverTime` |
| taskId | Long | Current task ID |
| resultVersion | Long | Current result version; current service implementation may leave it `null` |
| page | Integer | Current page number |
| pageSize | Integer | Page size, fixed `10` |
| total | Integer | Total candidate count in this check result |
| totalPages | Integer | Total pages in this check result |
| candidates | Array<Object> | Candidate list for current page (up to 10) |

`candidates[]` (`MatchCandidateView`) fields:

| Field | Type | Description |
|---|---|---|
| matchId | Long | Match record ID |
| memberId | Long | Candidate member ID |
| username | String | Username |
| nickname | String | Nickname |
| gender | String | Gender |
| heightCm | Integer | Height |
| annualIncomeCny | Long | Annual income |
| city | String | City |
| hobbyText | String | Hobby text |
| character | String | Personality text |
| abilityText | String | Ability text |
| photoUrls | Array<String> | Profile image URLs uploaded by the candidate |
| currentLocationText | String | Location text |
| currentLatitude | BigDecimal | Latitude |
| currentLongitude | BigDecimal | Longitude |
| rankScore | Double | Total score returned by `/check` |
| exactMatchCount | Integer | Exact matched condition count |
| exactConditionCount | Integer | Total exact condition count |
| confirmedViolationCount | Integer | Confirmed violation count |
| dailyExposureCount | Integer | Daily exposure count |
| profileUpdatedAt | DateTime | Profile last updated time |

`dating-cli check` guarantees `candidates[].photoUrls` as an array (empty array when no photos).

Reminder rule:

- The backend checks active tasks every 10 minutes.
- If email is configured, you will be notified via email when new candidates are identified, email can be configured through `dating-cli profile update [options]`. 

## 9. Result Commands

Mapped API: `POST /match-results/{matchId}/reveal-contact`

Input fields:

| Field | Type | Required | Description |
|---|---|---|---|
| matchId | Integer | Yes | Match record ID |

Output fields (top-level):

| Field | Type | Description |
|---|---|---|
| ok | Boolean | Command success |
| response.code | Integer | Business status code |
| response.message | String | Business message |
| response.data | Object | Contact payload |

Output fields (`response.data`):

| Field | Type | Description |
|---|---|---|
| matchId | Long | Match record ID |
| targetMemberId | Long | Target member ID |
| phone | String | Phone |
| telegram | String | Telegram |
| wechat | String | WeChat |
| signal_chat | String | Signal |
| line | String | Line |
| snapchat | String | Snapchat |
| instagram | String | Instagram |
| facebook | String | Facebook |

### 9.2 `dating-cli review <matchId> --rating <value> [--comment <text>]`

Mapped API: `POST /match-results/{matchId}/reviews`

Input fields:

| Field | Type | Required | Description |
|---|---|---|---|
| matchId | Integer | Yes | Match record ID |
| --rating | Integer | Yes | Rating, must be in `1..5` |
| --comment | String | No | Review text |

Output fields:

| Field | Type | Description |
|---|---|---|
| ok | Boolean | Command success |
| response.code | Integer | Business status code |
| response.message | String | Business message |
| response.data | Object | API payload (usually empty object) |

Business rule:

- The same user can review the same `matchId` only once.
